Introduction to State

Managing dynamic data in React components

What is State and Why We Need It

State is local, mutable data that belongs to a component and determines its behavior and rendering. It represents the component's current condition or "memory" that can change over time, allowing React components to create dynamic, interactive user experiences.

Core Characteristics

  • Component-specific: Each instance of a component has its own independent state
  • Mutable: Can be updated using special state setter functions
  • Preserved: Values persist between renders as long as the component stays mounted
  • Triggers renders: When state changes, React re-renders the component
  • Asynchronous: State updates may be batched for performance optimization
  • Private: State is fully controlled by the owning component
  • Transient: Lost when component unmounts (unless persisted elsewhere)

React components need state to:

  • Remember information between renders
  • Trigger re-renders when data changes
  • Reflect dynamic user interactions (clicks, form inputs, etc.)
  • Control UI elements that need to update dynamically
  • Store values that affect component rendering
  • Manage application flow and logic
  • Track asynchronous operations (loading states, API responses)
  • Implement component lifecycle behavior

Why State Matters

Without state, React components would be static and unable to respond to user interactions or data changes. State is what makes your UI interactive and dynamic. When state changes, React efficiently re-renders only the necessary parts of your UI through its reconciliation process. This makes React applications both responsive and performant.

State serves as a bridge between user actions and visual updates, creating the foundation for React's declarative programming model where UI is a function of the current state.

useState Hook Fundamentals

The useState Hook is the primary way to add state to functional components in React, enabling components to have their own state without writing a class:

Core Concepts

  • Declaration: const [state, setState] = useState(initialValue);
  • Array destructuring: Returns current state and a setter function
  • Initial value: Can be any valid JavaScript value (number, string, boolean, array, object, null)
  • Lazy initialization: Can accept a function: useState(() => computeInitialState()) that runs only on first render
  • State updates: Pass new value to setter (setState(newValue)) or function that computes new value (setState(prev => computeNewValue(prev)))
  • Batched updates: Multiple state updates in same event handler are batched in React 18+
  • Replacement, not mutation: States are replaced entirely, not merged or mutated

Best Practices

  • Use multiple useState calls for unrelated state values
  • Use a single useState object for closely related state values
  • Prefer functional updates (setState(prev => ...)) when updating based on previous state
  • Keep state minimal and compute derived values outside state
  • Avoid deep nesting when managing complex state structures

State vs Props

Understanding the differences between state and props is crucial for effective React development and data flow architecture:

State

  • Managed internally by component
  • Can be modified by component
  • Causes re-render when updated
  • Private to the component
  • Initialized and updated within component
  • Useful for data that changes over time
  • Lost on component unmount unless persisted
  • Can be passed down as props to children
  • Managed with useState or useReducer hooks

Props

  • Passed from parent components
  • Read-only (immutable)
  • Updates come from parent
  • Help create reusable components
  • Enable parent-child communication
  • Can include functions, objects, or primitives
  • Available on first render
  • Can have default values
  • Can be destructured for cleaner syntax

Working Together

Props and state often work together. A parent component can maintain state and pass it down as props to children. When state updates in the parent, the props in the children update automatically, causing them to re-render. This unidirectional data flow is a fundamental React pattern:

  • State becoming props: A component's state can become another component's props
  • Prop callbacks: Child components can receive functions as props to communicate back to parents
  • Derived state: Sometimes props can influence a component's internal state
  • State lifting: Moving state up the component tree for shared access
function Parent() {
  // State in the parent
  const [count, setCount] = useState(0);
  
  return (
    // Pass state as props to child
    <Child count={count} onIncrement={() => setCount(count + 1)} />
  );
}

function Child({ count, onIncrement }) {
  // Child receives props from parent
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={onIncrement}>Increment</button>
    </div>
  );
}

When to Use State

Not all data needs to be stored in state. Making smart decisions about what should be state is crucial for writing efficient React applications:

✅ Use State When

  • Data changes over time
  • User interactions affect the data
  • You need to preserve values between renders
  • Data influences what gets rendered
  • Data from asynchronous operations
  • Form input values and validation status
  • UI state like toggles, modals, and accordions
  • Navigation and routing state
  • Animation progress and status

❌ Don't Use State For

  • Data that can be calculated from props or other state
  • Data that doesn't affect rendering
  • Duplicated data from props
  • Constants that never change
  • DOM elements (use refs instead)
  • Cached/memoized values (use useMemo instead)
  • Previous values just for comparison (use useRef)
  • Direct copies of props without transformation
  • Values only used during render (regular variables)

State Location Matters

Choose the right component to hold each piece of state. Place state as close as possible to components that use it, but high enough to share it when needed. This is known as "lifting state up".

  • Local component state: For UI-specific state or data used only by one component
  • Lifted state: For data shared by multiple components but not the entire app
  • Global state: For data needed throughout your application (Context API, Redux, etc.)
  • Server state: For data that represents server-side resources (React Query, SWR)

State Immutability Principles

React relies on immutability for efficient rendering and predictable behavior. Always treat state as immutable and create new objects/arrays instead of mutating existing ones:

Why Immutability Matters

  • Predictability: Immutability makes state changes more predictable and easier to trace
  • Performance: React relies on reference changes to detect updates efficiently
  • Pure components: Enables React's optimization with memo, PureComponent, etc.
  • Time-travel debugging: Makes it possible to record and replay state changes
  • Concurrency: Prevents issues with React's concurrent rendering features
  • Undo/redo: Simplifies implementation of history features

❌ Incorrect: Direct Mutation

// Mutating objects directly won't work
const handleClick = () => {
  user.name = 'New Name';
  setUser(user); // React won't detect change!
};

// Mutating arrays directly won't work
const handleAdd = () => {
  items.push('New Item');
  setItems(items); // React won't detect change!
};

✅ Correct: Creating New References

// Creating new object with spread
const handleClick = () => {
  setUser({ ...user, name: 'New Name' });
};

// Creating new array with spread
const handleAdd = () => {
  setItems([...items, 'New Item']);
};

Immutability Helpers

Working with deeply nested state can be cumbersome with spread operators. Consider using immutability helper libraries:

  • Immer: Lets you write mutable code that produces immutable results
  • Immutable.js: Provides persistent immutable data structures
  • useImmer hook: Combines useState with Immer for elegant state updates
// Using Immer with useImmer hook
import { useImmer } from 'use-immer';

function TodoList() {
  const [todos, updateTodos] = useImmer([
    { id: 1, text: 'Learn React', done: false },
    { id: 2, text: 'Build an app', done: false }
  ]);
  
  const toggleTodo = (id) => {
    updateTodos(draft => {
      const todo = draft.find(t => t.id === id);
      todo.done = !todo.done;
    });
  };
  
  // This is immutable despite looking like direct mutation
  return null;
}

Event Handling

Responding to user interactions in React

Handling User Interactions

React uses a declarative approach to handle events by attaching event handlers to JSX elements. Unlike traditional DOM events, React events are named using camelCase syntax and passed as functions rather than strings.

Key Concepts

  • Declarative binding: Events are attached directly in JSX using props
  • camelCase naming: Events use camelCase naming convention (onClick instead of onclick)
  • Function references: Pass function references, not function calls
  • Arrow functions: Often used for inline handlers or when passing arguments
  • Method binding: Event handlers need correct 'this' context in class components

❌ Incorrect

// Don't call the function during render


// Don't use string references like in HTML

✅ Correct

// Pass function reference
<button onClick={handleClick}>Click</button>

// Or use inline arrow function
<button onClick={() => handleClick(arg)}>Click</button>

Event Object and Synthetic Events

React wraps native browser events in cross-browser compatible SyntheticEvent objects, providing consistent behavior across different browsers.

Key Properties

  • event.target: The DOM element that triggered the event
  • event.currentTarget: The element the event handler is attached to
  • event.preventDefault(): Prevents default browser behavior
  • event.stopPropagation(): Stops event bubbling up the DOM
  • event.type: The type of event (e.g., 'click', 'change')
  • event.nativeEvent: Access to the underlying browser event

Important Notes

  • SyntheticEvents are pooled and reused for performance in React
  • Event properties are nullified after the event callback is invoked
  • To retain event properties asynchronously, use event.persist() or save specific properties
  • React normalizes events so they work consistently across browsers

Event Delegation in React

React implements event delegation automatically for performance. Instead of attaching event listeners to every node, React attaches a single event listener to the root of the document and handles events from a centralized system:

  • Performance optimization: React manages event listeners at the root level
  • Automatic cleanup: No need to manually remove event listeners
  • Dynamic elements: Events work automatically on dynamically created elements
  • Synthetic event normalization: React normalizes events across browsers

Preventing Default Behavior

Browser events often have default behaviors (form submissions reload the page, links navigate, etc.). React allows you to prevent these behaviors using event.preventDefault().

Common Use Cases

  • Link clicks: Handle navigation in single-page applications
  • Context menus: Create custom right-click menus
  • Drag events: Implement custom drag and drop functionality
  • Key events: Override default keyboard behaviors

State Management Patterns

Advanced techniques for managing component state

Lifting State Up

When multiple components need to share and synchronize the same state, move that state to their closest common ancestor. This pattern is known as "lifting state up".

Key Benefits

  • Single source of truth: Maintains one authoritative state source
  • Synchronized data: All child components work with the same data
  • Simplified data flow: Parent passes state down, children trigger updates via callbacks
  • Clearer ownership: Parent component clearly owns and controls the shared state
  • Easier debugging: State changes are centralized and easier to track
function TemperatureInput({ scale, temperature, onTemperatureChange }) {
  const scaleNames = {
    c: 'Celsius',
    f: 'Fahrenheit'
  };
  
  return (
    <fieldset>
      <legend>Enter temperature in {scaleNames[scale]}:</legend>
      <input
        value={temperature}
        onChange={(e) => onTemperatureChange(e.target.value)}
      />
    </fieldset>
  );
}

function BoilingVerdict({ celsius }) {
  if (celsius >= 100) {
    return <p>The water would boil.</p>;
  }
  return <p>The water would not boil.</p>;
}

// Helper functions
function toCelsius(fahrenheit) {
  return (fahrenheit - 32) * 5 / 9;
}

function toFahrenheit(celsius) {
  return (celsius * 9 / 5) + 32;
}

function tryConvert(temperature, convert) {
  const input = parseFloat(temperature);
  if (Number.isNaN(input)) {
    return '';
  }
  const output = convert(input);
  const rounded = Math.round(output * 1000) / 1000;
  return rounded.toString();
}

function TemperatureCalculator() {
  // State is lifted to the parent component
  const [temperature, setTemperature] = useState('');
  const [scale, setScale] = useState('c');
  
  // Calculated values based on shared state
  const celsius = scale === 'f' ? tryConvert(temperature, toCelsius) : temperature;
  const fahrenheit = scale === 'c' ? tryConvert(temperature, toFahrenheit) : temperature;

  return (
    <div>
      {/* Child A can update shared state */}
      <TemperatureInput
        scale="c"
        temperature={celsius}
        onTemperatureChange={(t) => {
          setTemperature(t);
          setScale('c');
        }}
      />
      
      {/* Child B can also update the same shared state */}
      <TemperatureInput
        scale="f"
        temperature={fahrenheit}
        onTemperatureChange={(t) => {
          setTemperature(t);
          setScale('f');
        }}
      />
      
      {/* Child C just consumes the shared state */}
      <BoilingVerdict celsius={parseFloat(celsius)} />
    </div>
  );
}

State Colocation

State colocation is the principle of keeping state as close as possible to where it's used. This is often the opposite of lifting state up - when state is only needed by a specific component, it should be kept within that component.

Advantages

  • Reduced prop drilling: Eliminates passing props through multiple intermediate components
  • Improved performance: Fewer components re-render when state changes
  • Better code organization: Components are more self-contained and modular
  • Simpler mental model: Easier to understand which component owns what state
  • Improved maintainability: Changes to one component's state don't affect others

❌ Suboptimal: State Too High

function ParentList() {
  // State for all items plus expanded state
  const [items, setItems] = useState([]);
  const [expandedId, setExpandedId] = useState(null);
  
  return (
    <div>
      {items.map(item => (
        <ListItem 
          key={item.id}
          item={item}
          isExpanded={item.id === expandedId}
          onExpand={() => setExpandedId(item.id)}
        />
      ))}
    </div>
  );
}

✅ Better: Colocated State

function ParentList() {
  // Only items list is needed at this level
  const [items, setItems] = useState([]);
  
  return (
    <div>
      {items.map(item => (
        <ListItem key={item.id} item={item} />
      ))}
    </div>
  );
}

function ListItem({ item }) {
  // Expanded state is colocated here where used
  const [isExpanded, setIsExpanded] = useState(false);
  
  return (
    <div>
      <button onClick={() => setIsExpanded(!isExpanded)}>
        {isExpanded ? 'Collapse' : 'Expand'}
      </button>
      {isExpanded && <div>{item.details}</div>}
    </div>
  );
}

Controlled vs Uncontrolled Components

React offers two approaches to managing form inputs and other interactive elements: controlled and uncontrolled components. Each has specific use cases and trade-offs.

Controlled Components

  • Value source: React state
  • Updates via: onChange events
  • Data flow: Bidirectional
  • Validation: Real-time possible
  • Flexibility: High (can transform input)
  • Complexity: Higher

Uncontrolled Components

  • Value source: DOM itself
  • Access via: refs
  • Data flow: Form to component
  • Validation: Usually at submission
  • Flexibility: Limited
  • Complexity: Lower

When to Use Each

Use Controlled When:

  • Immediate validation is needed
  • Conditionally disabling submit button
  • Enforcing input format
  • Several inputs for one piece of data
  • Dynamic inputs

Use Uncontrolled When:

  • Simple forms with few fields
  • Integrating with non-React code
  • File inputs
  • Performance is a concern
  • Quick prototyping

Controlled Example 

function ControlledForm() {
  const [value, setValue] = useState('');
  const [error, setError] = useState('');
  
  const handleChange = (e) => {
    const newValue = e.target.value;
    setValue(newValue);
    
    // Real-time validation
    if (newValue.length < 3) {
      setError('Value must be at least 3 characters');
    } else {
      setError('');
    }
  };
  
  return (
    <div>
      <input
        value={value}
        onChange={handleChange}
      />
      {error && <p className="text-red-500">{error}</p>}
    </div>
  );
}

Uncontrolled Example 

function UncontrolledForm() {
  const inputRef = useRef();
  const [error, setError] = useState('');
  
  const handleSubmit = (e) => {
    e.preventDefault();
    const value = inputRef.current.value;
    
    // Validation at submission time
    if (value.length < 3) {
      setError('Value must be at least 3 characters');
    } else {
      setError('');
      // Process the form...
      console.log('Submitting:', value);
    }
  };
  
  // Return JSX for the form
  return (
    <div>
      <form onSubmit={handleSubmit}>
        <input ref={inputRef} defaultValue="default" />
        {error && <p>{error}</p>}
        <button type="submit">Submit</button>
      </form>
    </div>
  );
}

State Updates and Batching

Understanding how React processes state updates is crucial for avoiding common pitfalls and optimizing performance.

State Update Characteristics

  • Asynchronous: State updates may not be applied immediately
  • Batched: Multiple state updates in same event handler are grouped
  • Queued: Updates happen in the order they were called
  • Deferred: Actual DOM updates happen after all event handlers run
  • Closures: Updates may reference stale state values in closures

Functional Updates

Use functional updates when new state depends on previous state to avoid stale state references:

❌ Incorrect

// May cause issues with multiple updates
setCount(count + 1);
setCount(count + 1);

// Both updates use the same initial state
// So only one increment happens

✅ Correct

// Guaranteed to work with previous state
setCount(prev => prev + 1);
setCount(prev => prev + 1);

// Each update builds on the previous one
// Count will increase by 2

Automatic Batching

React batches multiple state updates within the same synchronous event handler to improve performance:

Batching Behavior

  • React 18+: All state updates are automatically batched (native events, promises, timeouts, etc.)
  • Prior to React 18: Only updates within React event handlers were batched
  • Performance benefit: Prevents multiple re-renders for multiple state updates
  • Mental model: Think of state updates as scheduled changes for the next render
function UserProfile() {
  const [name, setName] = useState('');
  const [age, setAge] = useState(0);
  const [address, setAddress] = useState('');

  function handleSubmit() {
    // These all batch into a single render
    setName(formData.name);
    setAge(formData.age);
    setAddress(formData.address);
    
    // React only performs ONE render with all updates
  }
  
  // In React 18, even these are batched
  async function handleFetch() {
    const response = await fetchUserData();
    
    // These updates will be batched in React 18
    setName(response.name);
    setAge(response.age);
    setAddress(response.address);
  }
}

Best Practices

  • Use functional updates when next state depends on previous state
  • Group related state in objects to update multiple values at once
  • Use useReducer for complex state logic with multiple sub-values
  • Avoid setting state inside useEffect cleanup functions
  • Remember that state updates in React 18 are always batched by default

Key Takeaways

  • State is for data that changes over time
  • Always treat state as immutable
  • Lift state up for shared data
  • Use functional updates for state changes
