Understanding JSX

JSX (JavaScript XML) is a syntax extension for JavaScript that looks similar to HTML but comes with the full power of JavaScript. It makes writing React components more intuitive and readable.

JSX in a nutshell

JSX looks like HTML, but it's actually a syntax extension for JavaScript that gets transformed into React.createElement() calls under the hood.

App.jsx
const
greeting = "Hello World";
// This is JSX
const
element = <h1 className="title">{greeting}</h1>;

JSX vs HTML: Key Differences

1
HTML Attributes Become camelCase

HTML:

onclick="handleClick()" class="button-primary" tabindex="1"

JSX:

onClick={handleClick} className="button-primary" tabIndex={1}
2
Expressions in Curly Braces
// Embedding JavaScript expressions
const
name = 'John';
const element = <h1>Hello, {name}!</h1>;
// Using JavaScript inside attributes
<img src={user.avatarUrl} alt={user.name} />
// Conditional rendering
<div>{isLoggedIn ? 'Welcome back!' : 'Please sign in'}</div>
3
Self-closing Tags Need a Slash

HTML:

<input type="text"> <img src="logo.png">

JSX:

<input type="text" /> <img src="logo.png" />
4
Must Return a Single Root Element

// Valid JSX with div wrapper:

<div>
<h1>Title</h1>
<p>Paragraph</p>
</div>

// Using React Fragments:

<>
<h1>Title</h1>
<p>Paragraph</p>
</>

Behind the Scenes: How JSX Works

JSX is transpiled to regular JavaScript by tools like Babel. When you write:

const element = <h1>Hello, world!</h1>;

It compiles to:

const element = React.createElement( 'h1', null, 'Hello, world!' );

This is why you need to import React in files that use JSX (for React versions before 17). React 17+ no longer requires this import thanks to the new JSX Transform.

Rendering Elements

React elements are the smallest building blocks of React applications. An element describes what you want to see on the screen. Unlike browser DOM elements, React elements are plain objects and are cheap to create.

The Rendering Process

1
Creating Elements

First, you create React elements using JSX or React.createElement()

const element = <h1>Hello, world</h1>;
2
Passing to Root

You then pass these elements to ReactDOM.render() or root.render() with a container DOM element

// React 18
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(element);

// React 17 and earlier
ReactDOM.render(element, document.getElementById('root'));
3
Virtual DOM Reconciliation

React creates a virtual DOM representation and compares it with the previous one

4
DOM Updates

React efficiently updates only the necessary parts of the real DOM

Example: Clock Component

This example shows how React re-renders elements when state changes:

import React, { useState, useEffect } from 'react';

function Clock() {
  const [date, setDate] = useState(new Date());
  
  useEffect(() => {
    const timerID = setInterval(() => {
      setDate(new Date());
    }, 1000);
    
    // Cleanup when component unmounts
    return () => {
      clearInterval(timerID);
    };
  }, []); // Empty dependency array means this runs once on mount
  
  return (
    <div>
      <h2>Current Time</h2>
      <p className="time">{date.toLocaleTimeString()}</p>
    </div>
  );
}

export default Clock;
How It Works:
  1. The Clock component initializes with current date in state
  2. The useEffect hook sets up an interval that updates the date every second
  3. Every time the state changes, React re-renders just the parts that need to change
  4. React is smart enough to only update the text content of the time paragraph
  5. When the component unmounts, the interval is cleared to prevent memory leaks

Important Concept: React Updates Are Immutable

React elements are immutable. Once you create an element, you can't change its children or attributes. An element is like a single frame in a movie: it represents the UI at a certain point in time.

To update the UI, you create a new element and pass it to root.render(), or you use state which triggers a re-render when it changes.

Functional vs Class Components

React supports two types of components: functional and class components. While both can achieve similar results, functional components are now the recommended approach with the introduction of Hooks in React 16.8.

Side-by-Side Comparison

Recommended
Functional Component
import React, { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
    </div>
  );
}
Legacy
Class Component
import React, { Component } from 'react';

class Counter extends Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }
  
  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <button onClick={() => 
          this.setState({ count: this.state.count + 1 })
        }>
          Increment
        </button>
      </div>
    );
  }
}
Functional Component Benefits
  • Simpler and more concise syntax
  • Hooks provide powerful state and lifecycle features
  • Easier to test and debug
  • Better performance (potentially) with optimizations
  • More aligned with modern JavaScript practices
  • Encourages code reuse and composition
Class Component Benefits
  • Familiar for developers from OOP backgrounds
  • Full access to lifecycle methods
  • Access to this.props and this.state
  • Required for error boundaries (as of React 17)

Modern React Best Practice

The React team recommends using functional components and hooks for new code. Hooks were introduced specifically to bring all the capabilities of class components to functional components, while enabling better patterns for reusing stateful logic between components.

// Custom hook example - reusable state logic
function useCounter(initialValue = 0, step = 1) {
  const [count, setCount] = useState(initialValue);
  const increment = () => setCount(count + step);
  const decrement = () => setCount(count - step);
  return { count, increment, decrement };
}

Props & PropTypes

Props (short for "properties") are how components receive data from their parent. They are the React equivalent of function arguments and enable components to be dynamic and reusable.

Understanding Props

Props Flow
  1. Parent component defines props as attributes
  2. Child component receives props as an object
  3. Props are read-only (immutable)
  4. Props flow downward (parent to child, never upward)
Key Prop Concepts
  • Props can be any valid JavaScript value
  • Props are readonly - never modify them!
  • Use state to manage changing values
  • Props enable component reusability

Sometimes called the "render prop" pattern, this advanced technique passes a function as children:

function Toggle({ children }) {

  • Content injection - Let parent components decide what to render inside children
  • Reusability - Create wrapper components that can be used with various content

Component Composition

Component composition is the practice of building complex components by combining smaller, more focused components. It follows the Unix philosophy: "Do one thing and do it well." This approach creates more maintainable, testable, and reusable code.

Basic Component Composition

Here's an example of building a complex UI using component composition:

function ProfilePage({ user }) {
  return (
    <Page>
      <Header>
        <Avatar src={user.avatarUrl} size="large" />
        <UserName>{user.name}</UserName>
      </Header>
      
      <Tabs>
        <Tab label="Posts">
          <PostList posts={user.posts} />
        </Tab>
        <Tab label="Photos">
          <PhotoGrid photos={user.photos} />
        </Tab>
        <Tab label="About">
          <UserBio bio={user.bio} />
        </Tab>
      </Tabs>
    </Page>
  );
}
Key Benefits
  • Readability: The code reads like HTML and clearly shows the component hierarchy
  • Reusability: Each component can be reused in other parts of the application
  • Testability: Smaller components are easier to test in isolation
  • Separation of concerns: Each component has a single, well-defined responsibility

Composition vs Inheritance

React strongly favors composition over inheritance for building component hierarchies. Almost anything you'd want to achieve with inheritance can be done more cleanly with composition.

From React Documentation:

"Props and composition give you all the flexibility you need to customize a component's look and behavior in an explicit and safe way. Remember that components may accept arbitrary props, including primitive values, React elements, or functions."

Component Naming Conventions

Good naming conventions for your components make your code more readable, maintainable, and professional. Consistent naming also helps team collaboration and makes your codebase easier to navigate.

Best Practices for Component Names

Do
  • Use PascalCase for component names (UserProfile)
  • Use descriptive, clear names (ProductList)
  • Use nouns or noun phrases for components (Button, ShoppingCart)
  • Add context prefixes for related components (UserProfile, UserSettings)
  • Name HOCs with "with" prefix (withAuth, withTheme)
  • Name custom hooks with "use" prefix (useFormInput)
Don't
  • Use lowercase for component names (button)
  • Use generic names (Component1, Page)
  • Use abbreviations that aren't universally understood
  • Use names that conflict with HTML elements (Div, Span)
  • Use extremely long names (UserDataDisplayComponentWithFormattedText)
  • Add "Component" suffix (ButtonComponent)

Component Creation Guide

Creating well-structured, reusable components is an essential skill for React developers. This step-by-step guide will help you create components that are easy to use, maintain, and test.

Component Creation Checklist

1

Plan Your Component's Responsibility

Define a clear, single responsibility for your component. Ask yourself:

  • What will this component display?
  • What functionality will it handle?
  • What data will it need?
  • Will it manage its own state?
  • Could it be broken down into smaller components?
2

Create the Basic Component Structure

Start with the simplest version of your component:

import React from 'react';

function ProductCard() {
  return (
    <div className="product-card">
      <h3>Product Name</h3>
      <p>$19.99</p>
    </div>
  );
}

export default ProductCard;
3

Define Props

Make your component dynamic with props:

import React from 'react';

function ProductCard({ name, price, imageUrl }) {
  return (
    <div className="product-card">
      {imageUrl && <img src={imageUrl} alt={name} />}
      <h3>{name}</h3>
      <p>${price.toFixed(2)}</p>
    </div>
  );
}

export default ProductCard;
4

Add State (If Needed)

Add state for interactive components:

import React, { useState } from 'react';

function ProductCard({ name, price, imageUrl }) {
  const [isExpanded, setIsExpanded] = useState(false);
  
  return (
    <div className="product-card">
      {imageUrl && <img src={imageUrl} alt={name} />}
      <h3>{name}</h3>
      <p>${price.toFixed(2)}</p>
      
      <button onClick={() => setIsExpanded(!isExpanded)}>
        {isExpanded ? 'Show Less' : 'Show More'}
      </button>
      
      {isExpanded && (
        <div className="details">
          <p>Additional product details here...</p>
        </div>
      )}
    </div>
  );
}
5

Add Event Handlers and Props

Make the component interactive:

      <div className="buttons">
        <button onClick={() => setIsExpanded(!isExpanded)}>
          {isExpanded ? 'Show Less' : 'Show More'}
        </button>
        
        <button 
          onClick={() => onAddToCart({ name, price })}
          className="add-to-cart"
        >
          Add to Cart
        </button>
      </div>
      
      {isExpanded && (
        <div className="details">
          <p>Additional product details here...</p>
        </div>
      )}
    </div>
  );
}

Component Development Best Practices

  • Single Responsibility: Each component should do one thing well
  • Keep Components Small: Aim for under 250 lines of code
  • Lift State Up: Move shared state to the nearest common ancestor
  • Use Composition: Favor composition over inheritance
  • Write Pure Components: Avoid side effects in render functions
  • Use Default Props: Provide sensible defaults when possible
  • Document Your Components: Add comments for complex logic
  • Don't Repeat Yourself: Extract reusable logic into custom hooks
Final Component Example
import React, { useState, memo } from 'react';
import PropTypes from 'prop-types';
import './ProductCard.css';

function ProductCard({ product, onAddToCart }) {
  const [isExpanded, setIsExpanded] = useState(false);
  
  // Handle click on the add to cart button
  const handleAddToCart = () => {
    onAddToCart(product.id);
  };
  
  return (
    <div className="product-card">
      {product.image && (
        <img 
          src={product.image} 
          alt={product.name} 
          className="product-image" 
        />
      )}
      
      <div className="product-info">
        <h3 className="product-title">{product.name}</h3>
        <p className="product-price">${product.price.toFixed(2)}</p>
        
        {isExpanded && (
          <p className="product-description">{product.description}</p>
        )}
      </div>
      
      <div className="product-actions">
        <button 
          onClick={() => setIsExpanded(!isExpanded)}
          className="btn btn-secondary"
        >
          {isExpanded ? 'Show Less' : 'Show More'}
        </button>
        
        <button 
          onClick={handleAddToCart}
          className="btn btn-primary"
          disabled={product.outOfStock}
        >
          {product.outOfStock ? 'Out of Stock' : 'Add to Cart'}
        </button>
      </div>
    </div>
  );
}

ProductCard.propTypes = {
  product: PropTypes.shape({
    id: PropTypes.string.isRequired,
    name: PropTypes.string.isRequired,
    price: PropTypes.number.isRequired,
    description: PropTypes.string,
    image: PropTypes.string,
    outOfStock: PropTypes.bool
  }).isRequired,
  onAddToCart: PropTypes.func.isRequired
};

export default memo(ProductCard);

Component Best Practices

  • Keep components focused on one responsibility
  • Use clear, descriptive names following PascalCase convention
  • Start component names with a capital letter (Button, Card)
  • Use functional components with hooks instead of class components
  • Document your components with PropTypes or TypeScript
  • Extract repeated code into separate, reusable components
  • Keep components under 250 lines of code when possible
  • Use composition to build complex UIs from simple components
